# 将来用ルール（検討済み・現在不要）

このディレクトリには、**検討したが現在のKomonでは不要と判断したルール**が格納されています。

## 🎯 このディレクトリの目的

### 1. 検討の履歴を残す

「検討して不要と判断した」のか「そもそも検討していない」のかを明確にする。

### 2. 将来の判断材料

将来、同じ機能が必要になった際に：
- 過去の検討内容を参照できる
- ゼロから考える必要がない
- 判断基準が明確

### 3. 他プロジェクトへの適用

Komonでは不要でも、他のプロジェクトでは必要な場合がある。

---

## 📋 検討済みルール一覧

### 1. API連携ルール（api-integration-rules.md）

**検討日**: 2025-11-29  
**判断**: 現在不要  
**理由**:
- Komonは外部API連携を行わない
- Slack/Email通知は単純なHTTP POSTのみ
- 複雑なリトライ戦略は不要

**将来必要になるケース**:
- 外部監視サービスとの連携
- クラウドAPIとの統合
- 複雑な認証が必要な場合

**再検討タイミング**:
- 外部API連携機能を追加する際

---

### 2. データベース操作ルール（database-rules.md）

**検討日**: 2025-11-29  
**判断**: 現在不要  
**理由**:
- Komonはファイルベースのデータ管理
- SQLiteすら使用していない
- トランザクション管理は不要

**将来必要になるケース**:
- 大量データの管理が必要
- 複雑なクエリが必要
- 複数プロセスからのアクセス

**再検討タイミング**:
- データ量が増えてファイルベースが限界になった際
- 複数マシンでのデータ共有が必要になった際

---

### 3. パフォーマンス基準（performance-standards.md）

**検討日**: 2025-11-29  
**判断**: 現在不要  
**理由**:
- Komonは軽量ツール（数秒で完了）
- パフォーマンス問題が発生していない
- 厳密な基準は不要

**将来必要になるケース**:
- 大量データ処理が必要
- リアルタイム性が求められる
- ユーザーから遅いという報告

**再検討タイミング**:
- 実際にパフォーマンス問題が発生した際
- 処理時間が5秒を超えるようになった際

---

### 4. セキュリティガイドライン（security-guidelines.md）

**検討日**: 2025-11-29  
**判断**: 現在不要  
**理由**:
- Komonはローカル実行のみ
- 外部からのアクセスなし
- 機密情報は環境変数管理（既存ルールで十分）

**将来必要になるケース**:
- Web UIを追加
- マルチユーザー対応
- 外部公開する機能を追加

**再検討タイミング**:
- ネットワーク経由でのアクセスを許可する際
- 複数ユーザーで使用する際

## 🎯 設計思想

### なぜ削除しないのか？

1. **検討の履歴を残す**
   - 「検討して不要」と「検討していない」は全く違う
   - 将来の判断材料になる

2. **再検討のコストを下げる**
   - ゼロから考える必要がない
   - 過去の検討内容を参照できる

3. **他プロジェクトへの適用**
   - Komonでは不要でも、他では必要
   - テンプレートとして使える

### なぜアクティブ化しないのか？

1. **Komonをシンプルに保つ**
   - 不要なルールは読み込まない
   - Kiroの処理が軽量

2. **YAGNI原則**
   - You Aren't Gonna Need It
   - 必要になってから実装

3. **メンテナンスコストの削減**
   - 使わないルールは更新不要
   - 実際に使う時に最新化

## まとめ

- **削除しない**: 検討の履歴として残す
- **アクティブ化しない**: 現在は不要
- **将来の判断材料**: 必要になったら参照

このアプローチにより、「検討済み・現在不要」と「未検討」を明確に区別できます。
